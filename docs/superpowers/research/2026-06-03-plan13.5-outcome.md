## Architecture summary

**`rust-s3` instead of `aws-sdk-s3`.** Single-purpose S3 protocol client, ~30 transitive deps vs ~150 for the AWS SDK. Works with every S3-compatible provider (AWS S3, MinIO, Cloudflare R2, Backblaze B2, Wasabi, Hetzner, etc.) which matches knot's self-hosted audience better than IRSA-out-of-the-box would.

**`S3Store` API rewritten** to wrap `s3::Bucket`. New constructor surface:

```rust
S3Store::from_env(bucket, region, endpoint, prefix)  // reads AWS_* env
S3Store::new(bucket, region, endpoint, prefix, creds) // explicit creds
```

knot-server never touches the `s3` crate directly — `S3Store::from_env` keeps the SDK contained in `knot-storage`.

**Runtime selection in `main.rs`:** after `AppState::with_pool`, check `cfg.blob_backend == "s3"`. If yes, validate `cfg.s3_bucket`, build `S3Store::from_env(...)`, override `state.blob_store`. If no, leave the default (`PgBytesStore` set by `with_pool`). The log line emits the active backend at INFO level.

**Helm:** `blob.backend = postgres | s3` (default `postgres`). When `s3`, the chart writes `KNOT_BLOB_BACKEND` + `KNOT_S3_*` config to the ConfigMap and `envFrom`s `blob.s3.existingSecretName` so credentials come from a separately-managed Secret with keys `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` / (optional) `AWS_SESSION_TOKEN`.

## What was non-obvious

**Image size impact is much smaller than expected.** Pre-plan estimate was +25-30 MB for the AWS SDK. Actual after switching to rust-s3: **+1.8 MB** (19.5 → 21.3 MB). rust-s3 pulls hyper + reqwest (already transitive deps for OIDC discovery and OTLP) plus a thin S3-protocol layer; almost nothing new in the dependency graph.

**rust-s3's `Bucket::new` returns `Box<Bucket>`.** My first draft wrapped it in `Box` again, hit "expected Bucket, found Box<Bucket>" from rustc. Fixed by just letting the SDK's boxed return value flow through.

**`Region::from_str_default` doesn't exist.** Used `Region::from_str(&region).unwrap_or(Region::UsEast1)`. The crate uses standard `FromStr`.

**Path-style addressing is non-negotiable for S3-compat providers.** Called `.with_path_style()` unconditionally — native AWS supports it too and most non-AWS providers (MinIO, R2 with custom endpoints) require it.

**Hiding the SDK is worth doing.** First attempt had main.rs call `s3::creds::Credentials::default()` directly, which would have forced knot-server to take a dep on the rust-s3 crate. Refactored to `S3Store::from_env(...)` so the SDK stays a knot-storage implementation detail. Saves one Cargo.toml entry and one churn vector if we ever swap the SDK again.

## What's still deferred

- **IRSA / instance-profile credentials.** rust-s3's `Credentials::default()` reads env only. EKS users with IRSA need to inject env vars via a sidecar or webhook (eks-iam-injector etc.). A future plan could add STS support directly.
- **Multipart uploads.** Single-PUT works up to multi-GB on most providers; the 10 MB cap from the server makes this moot.
- **Pre-signed URLs.** Defer to a CDN plan.
- **Lifecycle / TTL on the bucket side.** That's a bucket-policy concern, not a knot concern.
- **MinIO sidecar in dev compose.** Explicitly out per user preference; real S3 is the integration env.
