# Comments & Mentions Implementation Plan (Plan 19)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development (recommended) or superpowers:executing-plans.

**Goal:** Inline comment threads anchored to text in the editor, @mentions resolving to workspace members, emoji reactions, resolve/unresolve. Persisted in Postgres, joined to the live editor via Yjs `RelativePosition` for stable anchoring under concurrent edits.

**Architecture:**

- **Storage** — Postgres (per user decision). Two tables:
  - `comments(id, doc_id, thread_id, parent_id NULL, author_id, body, position_y BYTEA NULL, anchor_text TEXT NULL, created_at, updated_at, resolved_at NULL, deleted_at NULL)`. `thread_id` is the id of the thread root (self for root comments). `parent_id` is the immediate parent — for v0.1 we keep threads one-level (parent = thread root or null). `position_y` is the Yjs-encoded `RelativePosition` for anchored threads; null for whole-doc threads. `anchor_text` is a snippet of the selection at creation time (fallback when the position drifts).
  - `comment_reactions(comment_id, user_id, emoji, created_at, PK(comment_id, user_id, emoji))`.
- **Mentions** — body is stored as plain text. The frontend parses `@username` patterns client-side at render time and links them. Server-side, on POST/PATCH the mentioned user_ids are extracted and inserted into a join table OR simply emitted as a one-shot `mentions: [user_id, ...]` field for the in-app toast bridge (chosen: one-shot via WS broadcast to the workspace; persisted notification system is Plan 21).
- **CRDT anchoring** — Tiptap + y-prosemirror's `absolutePositionToRelativePosition` / `relativePositionToAbsolutePosition` plus `Y.encodeRelativePosition` / `Y.decodeRelativePosition`. The frontend converts a selection's start to a `RelativePosition`, encodes to Uint8Array, base64s it, and sends it as the `position_y` body. On render the panel decodes + resolves to an absolute position; if the resolution fails (text deleted), fall back to `anchor_text` as the title.
- **Reactions** — server endpoints; client renders an emoji row under each comment with a `+` button that shows a tiny picker (👍 🎉 ❤️ 🚀 👀 🙏 hard-coded for v0.1).
- **Resolve** — server endpoint toggles `resolved_at`. UI hides resolved by default; "Show resolved" toggle in the side panel surfaces them.
- **ACL** — Owner or Editor on the doc to create/reply/react/resolve. Viewer can read but not write. Author can edit/delete their own comments. Workspace Owner can delete any comment.

**Predecessor:** Plan 20 (HEAD `b9ab410`).

**Out of scope:**
- **Multi-level reply threading.** Replies attach to thread root for v0.1.
- **Persistent notifications table + bell UI.** In-app toast on @mention; persistent notifications are Plan 21.
- **Email notifications on mention.** Plan 18 + Plan 21.
- **Markdown formatting in comment bodies.** Plain text.
- **Comment edit history.** No edit log; PATCH overwrites.
- **Reactions beyond the fixed emoji set.** Free-text emoji input is a follow-up.
- **Activity feed.** Separate plan.

## Conventions

- **Body length:** 4 KB cap. Past that, 413.
- **Anchor encoding:** `position_y` is base64-encoded `Y.encodeRelativePosition` output. Server treats it as opaque bytes.
- **Anchor fallback:** when the frontend can't resolve a `RelativePosition` (text deleted), it shows the `anchor_text` snippet + a "Removed" label. Click still scrolls to the thread's position in the sidebar.
- **Mention parsing:** the comment body uses literal `@<display_name>` with display-name resolution at render time. Server extracts mentions via a regex on POST/PATCH and includes them in a `mentions: [{ user_id, display_name }]` field on the response.
- **Mention broadcast:** when a comment with mentions lands, the server broadcasts a small `{ type: "mention", doc_id, comment_id, mentioned_user_ids }` payload over the existing CRDT bus's `LISTEN/NOTIFY` channel. Connected clients of the mentioned users surface a toast. Out-of-band users see nothing for v0.1.
- **Reactions:** the emoji set is hard-coded `["👍","🎉","❤️","🚀","👀","🙏"]`. Server validates against the list (415 if invalid).
- **Resolution:** resolving a thread sets `resolved_at` on the thread root only. Replies are read-only after resolution.

  thread_id    UUID NOT NULL,        -- self for root, parent thread root for replies

  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  resolved_at  TIMESTAMPTZ NULL,      -- set on thread root only
  deleted_at   TIMESTAMPTZ NULL
);

CREATE INDEX comments_thread_idx     ON comments(thread_id) WHERE deleted_at IS NULL;

  comment_id   UUID NOT NULL REFERENCES comments(id) ON DELETE CASCADE,
  user_id      UUID NOT NULL REFERENCES users(id),
  emoji        TEXT NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (comment_id, user_id, emoji)
);

## Task 9: anchor.ts

`web/src/features/comments/anchor.ts`:

```ts
import * as Y from "yjs";
import { absolutePositionToRelativePosition, relativePositionToAbsolutePosition }
  from "y-prosemirror";

// Encode an absolute ProseMirror position as a base64 RelativePosition.
export function encodeAnchor(view, ydoc: Y.Doc, mapping, absPos: number): string {
  const rel = absolutePositionToRelativePosition(absPos, ydoc.getXmlFragment("default"), mapping);
  const bytes = Y.encodeRelativePosition(rel);
  return btoa(String.fromCharCode(...bytes));
}

// Decode + resolve a base64 RelativePosition back to an absolute position.
// Returns null when the underlying text has been deleted.
export function decodeAnchor(view, ydoc: Y.Doc, mapping, b64: string): number | null {
  const bytes = Uint8Array.from(atob(b64), c => c.charCodeAt(0));
  const rel = Y.decodeRelativePosition(bytes);
  const abs = relativePositionToAbsolutePosition(ydoc.getXmlFragment("default"), view.state.doc, rel, mapping);
  return abs == null ? null : abs.pos;
}
```

(Adapt signatures to what `y-prosemirror` actually exposes — the mapping object name may differ.)

Add vitest unit tests covering round-trip encode/decode.
